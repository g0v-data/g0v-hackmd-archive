## g0v Keng of Conf 坑抗交流聚會

因為社群沒有人的特性，會發生一些意料之外或意料之內的事情，甚至也會發生「最大的問題還是人」的狀況。因為耳聞不少 g0v 社群發生的大小事，g0v 揪松團想舉辦坑主小聚（a.k.a. g0v Keng of Conf 簡稱「坑抗」），希望可以給不同階段的坑主（初坑、中坑或深坑）一些經驗分享或共同討論解決方式。

:::spoiler 20200202 活動資訊
- 活動基本資訊
    - 時間：2020/02/02 (Sun.) 14:00 - 18:00
    - 地點：NPOhub 4F 廚房
    - 主持人：g0v 揪松團 jothon
    - 備註：本活動採邀請制。請洽 g0v 揪松團。 `g0v Slack #jothon`
- 活動流程構想
    - 自我介紹（如何稱呼、投入的專案、最新進度）。
    - 提出問題、想解決的事項。
        - 寫 `便利貼` 說明自己的問題（便利貼請留下稱呼，寫想討論的議題 or 報名時註記可能的問題 or 事前蒐集於此頁面下方）。

        - 每一組有一位小組主持人，依照該小組的題目與成員們，著手討論、想辦法在這個課題上有所進展。
    - 成果分享
        - 每一個小組向大家分享各小組討論的成果，並請其他參與者提問。
:::

## A.採用「面對這個問題，目前我們觀察到是這樣」的紀錄方式，整理坑主們在意的討論重點

### (1) 一個專案中有哪些角色？
- 專案小組 Core Team：特點是 穩定明確
    - 坑主 
        - 提出專案構想，推動專案進展
    - PM (Project Manager)
        - 了解專案技術，建立專案藍圖
    - PC (Project Coordinator)
        - 串起專案參與者
    - 職工 (聘僱執行)
- 專案參與者 Participants：特點是 流動協作
    - 持續貢獻者
    - 隨緣貢獻者
    - 多元技能：設計師、程式開發、文案撰寫...等
- 民間專案常見關聯領域諮詢角色：尋求 可信任的單位
    - 專案有帳務處理需求
    - 專案遇到的法律諮詢
    - 專案有募款募資的需求
    - 對於專案實質議題能提供諮詢意見的團體
    - 其他

### (2) 「專案/坑」的階段
- 新坑階段：提出專案構想，找出專案的工作方向，執行短期可行工作
    - 可以參考「提案模板 / 引導式課程 / 表格」來開展專案
        - https://g0v.hackmd.io/@jothon/keng/https%3A%2F%2Fg0v.hackmd.io%2Fa_sXd-DRSLmJhhd61sc_ag
    - 可以把提案構想，寫成專案構想文件，如運用 g0v.hackmd
    - 可以加入 g0v slack 頻道，分享專案構想文件
        - 社群成員可以推薦相近專案，例如既有的 slack 頻道，既有的專案網站或文件
        - 若過往沒有相關的專案，也推薦新坑主設立此專案的 slack 公開頻道
    - 可以參加兩個月一次的 g0v 黑客松活動，進行現場 3 分鐘提案
        - 活動前可以參加前一晚的松前哈拉活動，聊聊提案構想
        - 黑客松當天
            - 上午進行現場提案
            - 集思廣益，蒐集有興趣的參與者對於專案構想的執行方向
    - 提出專案的工作方向
        - 例如採用短期可行的、中長期再看看的分類方式，找出短期大家想執行的事項
    - 執行專案的工作方向
        - 依照專案藍圖逐步推動專案進度
        - 以下是一些這個階段會需要評估的事項，屬於專案的基礎建設
            - 取名子
            - Domain Name 的管理方式
            - 爬蟲伺服器、資料庫，該如何選擇
            - 設置對外聯絡信箱，例如採用 googlegroups 
- 長期坑第一階段：專案小組現身、協作機制成形
    - 形成「專案小組 Core Team」
        - 核心工作小組成員彼此討論的非公開頻道或溝通工具
        - 設置對外聯絡信箱，例如採用 googlegroups 
        - 經費收支方式
            - 專案若要設立捐款，評估採用什麼方案
            - 支出流程
    - 透過「專案架構」來整理工作需求
    - 有階段專案目標，且透過開源協作來更新與擴充
        - Slack 公開頻道
        - GitHub
        - Hackmd-bookmode
        - Figma
    - 「專案參與者 Participants」可依循專案採用的協作機制
    - 申請 Google 雲端空間，例如諮詢 OCF
    - 專案入口
        - hackmd-bookmode
        - 於 Aray 或 Badge 建立專案頁面
        - 建立網站
- 長期坑第二階段：誠信當責、財源有道
    - 將事務嘗試分擔分工
    - 面對「大松，小松，週會/例會，演講與曝光，對外合作」都能善用各類場域功效，促進專案進展充實
    - 專案社群治理有所依據，對外代表原則都有方案
    - 有「危險防止機制」(成果誤刪、個資外洩、個別代表混淆大眾視聽、募款財務帳務責信..等)
    - 與「關聯領域諮詢角色 (帳務、法律諮詢、募款募資)」建立可信任的合作模式與諮詢管道
        - 申請 OCF 社群專案建立募款金流系統
    - 申請相關專案資源
        - 如何申請 HackMD 公益贊助？https://hackmd.io/Kr3cdmgST1CeUcd_a8ZQVw
- 法人化 (社會企業或非營利組織)
    - (需再歸納相關經驗)

### (3) 需要以「專案架構」來整理工作
- 招不到人、專業性質太高，工作不好切割
    - 應該要把文件（read me）寫好，方便別人跳坑或接手。
    - 若專案架構大，把想做的切塊，訂好中間介面讓大家好跳坑。
    - 可能需要 PM 或是架構師的角色幫忙。
        - 這部分有沒有什麼樣的專案案例？ 
            - disfactory 
            - 政治獻金 OCR
        - Questions I ask to scope a new project in g0v hackathon (in 1–2 hr) by Yun Chen 
            - https://aelcenganda.medium.com/questions-to-scope-a-new-project-in-g0v-hackathon-38a1189c391c
            - What can be done in 1 day? (in the hackathon)
            - What can be done in 2 weeks?(first sprint)
            - What MVP can we build in 2–3 months?(one milestone)
    - 以新坑來說，推薦新提案者前一天參加松前哈拉，能檢視提案是否有問題，或是哪邊可以更好說明。
- 從影響力的角度來擬定
    - 要怎麼知道這個專案與功能開發做下去，是否能夠，以及是如何能夠，變現於議題的推動上？
    - 政府公部門政策與業務，社群專案，兩者之間的互動關係？以及是否能推動政府再採用開源時能夠有回饋的概念。
        - 可以請長期參與者幫專案預判，若要跟政府合作，要瞭解是如何構思專案進度，以及怎麼聯繫上窗口，或是有沒有可以參考的民間專案影響政策措施的經驗。
        - 多數 g0v 的共識應該是希望專案能夠推動並改進，較難去說從政府方得到回饋。
        - 不希望 g0v 變成有裙帶關係的感覺（因為在 g0v 認識高層，而取得標案或得到好處，但也不一定能夠回饋給開源專案的所有貢獻者）。
        - 與公部門協力的專案會有卡到上班時間的問題。[name=ky]
        - 前朝政績容易人亡政息。[name=ky]

### (4) 談專案的參與者、核心小組、招募人力

- 參與者說要做到但沒做到（或是能力不足以勝任） 
    - 一般專案運作中，為了留住新人，可能會花費更多時間去教導好的程式習慣 or 協作觀念及模式，將 5-6 分的品質提升到 7 分，剩下只能坑主自己補。
    - 沒有背景去判斷這個人寫出來的東西好不好，至少要確認這個人短時間不會跑掉，可以幫你修改東西。
    - 若牽扯到聘僱、薪水，即使是社群人，也應該說清楚要達到的績效及目標，若是擺爛勿心軟，請找人接手。

- 容易被亂入
    - 依照不同的專案／工作小組有不同的做法，可以強硬的請他離開，也可以冷處理。使用 Slack 的一大優點：已讀不回

- 沒有坑主、過度仰賴坑主。
  - 找到 core team 是必要的，三個人為佳，可以輪替。
  - 參與者可來去、但 core team 要穩定，不能每次決策的人數都不同
  - 文件很重要，若有文件比較好繼續存續，每個人都可以加入繼續（同樣也很可能被其他人直接拿去講說，我就是坑主）。
  - 應該要有意識地做接班的工作
  - 在貢獻之前有個承諾期間或許可以避免過度無止盡的消耗。（ex：任期一年 or 只有在固定時間範圍內貢獻）
  
- 參與的目的不同。
    - 有些人是新手想在這邊訓練變成工程師可能就比較不適合，但社群協作的模式很值得學習。
    - 雙月大松對於長期參與者而言變成社交場所非工作場所，若期待更多人參與，要多建立起新參者的連結，主動找他們聊天推坑。
    - 大松旅行團復活（？

- 調整方向時如何維繫社群參與度
    - 社群的自由度也在此，若要轉方向，誰有權責去說要轉向？但其實只要主動說要做，跟大家協調，確認大家都同意，就可以轉向。

- 有人可以享樂，卻沒有人要扛責任
    - 社群的外部成本，但是可以多跟大家提倡社群的環境是前人堆築起來的，請共同維護。

- 不公開 SNS 群組的界線（Line、Messenger 等）
    - 坑主自律、回到 slack
    - 非公開群組不要討論專案，不要有因專案得到的資源、資訊在其中流通

- 有人出來隨便代表社群（專案） -> 代表性問題
    - 回信權限：至少有一定程度的參與才能回信，確保回信內容不偏離現實
    - 要有 core team，有明確穩定公開的名單，若有牽扯到 Funding 或是錢更是如此。

### 專案要如何建立頻道？Slack, Disord 如何評估？

專案的訊息溝通模式，整理不同方案如下：
- (1) 於 g0v slack 開設單一頻道
- (2) 於 g0v slack 開設系列頻道，如 da0 採用 da0-xxx
- (3) 自建 Slack，如 島島阿學、選前大補帖
    - 此方案可選擇 Slack 其中一個頻道與 g0v slack 的指定頻道，同步訊息，相關工具請洽 g0v slack #bridge 頻道
- (4) 自建 Discord，如 Cofacts、足跡地圖
    - 此方案可選擇 Discord 其中一個頻道與 g0v slack 的指定頻道，同步訊息，相關工具請洽 g0v slack #bridge 頻道
- (5) 自建 FB Group，如 資料申請小幫手
- (6) 其他

對於不同方案，初步歸納一些差異
- 各方案，能否全文備份頻道內容？
    - 感謝 Ronny 將 g0v slack 公開頻道全文備份
- 能否裝設 app、機器人，以及數量是否有上限
    - g0v slack app 數量有上限 
    - 常見用途：
        - 監測並通報網站服務狀況
- 與 g0v slack 之間的訊息，是否有同步方案
    - 如 Cofacts Discord-general 與 #cofacts 同步
- 專案貢獻者群體，對於該訊息工具使用的順手程度
    - 例如 足跡地圖，蠻多參與者較熟悉 discord 而非 slack
- 各方案，通常會另開一個 非公開頻道 用於核心小組使用

其他
- 上述內容，可以整理成矩陣圖
- 開一個共編簡報，來製作矩陣圖

### 如何思考開源專案、g0v 專案的 CRM 方案？

觀察：
- chewei 提問動機來自於，去年開始參加 好植地 (盤點用地推動植樹) 專案推動，一方面有建立公開協作的 素材與知識庫；同時，隨著專案的推動，也會接觸到許多公私部門關心該專案主題的窗口，交流部門意見、CSR 企劃構想 ..等，所以當下的需求變成是，希望建構「能夠保護個資的 CRM，且與 公開協作頁面 (目前採用 Airtable) 的內容相關聯」
- 就這個需求回想起來，參與公有地與閒置公有資產活化等專案主題，應該也需要建立 CRM (或類似的內容)，2014 年至今自己與其他參與者，不太可能完整記得過去曾與哪些部門互動過。
- 猜想，每個專案似乎都有這樣的名單。
- 如何思考開源專案、g0v 專案的 CRM 方案？[討論串](https://g0v-tw.slack.com/archives/C0385B90D/p1612008201028000) [name=chewei]
    - CRM 資料，也可能是僅由 專案對外收發信者可以閱覽編輯

初步歸納建議：
- 專案的坑主與核心投入者，可以設置 GoogleGroup 信件群組，用於對外聯繫，也算是分擔收發信工作、備份信件記錄
- 長期坑的坑主們，可以建立 CRM (工具例如 Airtable) 瀏覽權限設定為該專案的坑主與核心投入者，以方便留下人員資料，便於其他專案成員可接手
