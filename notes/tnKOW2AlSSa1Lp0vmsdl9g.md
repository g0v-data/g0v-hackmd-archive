# 長期維運評估
開源的成果如何持續維運？

1. 前提：專案通過POC且有MVP了。
    - 數位解方/實體解方?
        - 軟體即服務 (SaaS) 服務提供者透過網際網路遞送軟體及應用程式。 使用者透過 Web 或供應商 API 存取它們。
        - 平台即服務 (PaaS) 提供雲端環境以供使用者建置與供應應用程式的服務提供者。 提供者供應基礎架構。
        - 基礎架構即服務 (IaaS) 供應商提供客戶透過隨用隨付制存取位在雲端的儲存容量、網路功能、伺服器及其他運算資源。
        - 手能觸及的實體，例如桌遊。開源的規則、開源的授權印製與改作。
        - 紙本...
        - 影片
    
    - 數位
        - APP
        - LINE Bot
        - Website
        - IaaS、 PaaS 及 SaaS 
        - File (例如PDF、Excel)
        - Link
        - Airtable
        - [sheet2site](https://www.sheet2site.com/)
    
    - 雲端/地端
        - 7/24 ?
        - 在我電腦上是活著的
        - 在我Docker/VM上是活著的 
        - 在NAS上是活著的
        - 在storage上是睡著的
        - 在歷史上是活著的

    - DevOps「開發」「測試」「維運」循環模式
        - 永動循環/持續整合/持續交付
        - 當循環已死
        - 你不動，我不動
    
    - Site Reliability Engineering (SRE)
        - 文化和心態上
        - 可靠性? 服務可以掛掉多久？
